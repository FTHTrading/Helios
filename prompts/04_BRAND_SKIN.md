# BRAND SKIN

> The Brand Skin changes the clothes. It never changes the body.

This prompt defines what is allowed to change between brands and provides the exact override specifications for each brand. Everything not listed here stays shared. No exceptions.

---

## When To Use This Prompt

- When deploying the shared shell for a specific brand
- When adjusting tone, copy, or visual emphasis for a brand
- When configuring the AI rail voice for a brand
- When naming features or modules for a brand's audience
- When creating marketing or onboarding content for a brand
- **Never** when building components, architecture, or core logic — that's the Builder's domain

---

## What The Brand Skin Controls

Exactly 6 things. Nothing more.

| Layer | What Changes | What Stays |
|-------|-------------|------------|
| **1. Palette** | Colors, gradients, glass tints, glow colors | Color roles (primary, secondary, accent, surface, text) |
| **2. Emphasis** | Which font gets prominence, weight distribution | Font families (Inter + EB Garamond), type scale |
| **3. Copy Tone** | Formal ↔ casual spectrum, vocabulary level | Information architecture, content structure |
| **4. Module Naming** | What features are called in the UI | Feature functionality, API endpoints, data models |
| **5. Voice Personality** | AI rail character, greeting style, suggestion tone | AI capabilities, context detection, action triggers |
| **6. Trust Emphasis** | Which proof surfaces get visual priority | Proof mechanics, verification flow, audit trail |

## Brand Skin Implementation

### CSS Theme File

Each brand gets exactly one theme file that overrides sovereign tokens:

```css
/* static/css/themes/[brand].theme.css */

:root {
  /* Only override color tokens */
  --sv-color-primary: [brand primary];
  --sv-color-secondary: [brand secondary];
  --sv-color-accent: [brand accent];
  --sv-glass-tint: [brand tint];
  --sv-glow-color: [brand glow];
  
  /* Only override emphasis tokens */
  --sv-font-headline: [brand headline font];
  --sv-font-weight-headline: [brand weight];
}
```

### Configuration Object

Each brand's non-visual settings come from a config object:

```python
BRAND_SKINS = {
    "helios": {
        "module_names": {
            "dashboard": "Command Center",
            "wallet": "Treasury",
            "referral": "Network Growth",
            "staking": "Energy Commitment",
            "rewards": "Yield",
            "certificate": "Proof of Participation"
        },
        "voice": {
            "tone": "analyst",
            "formality": 0.8,      # 0 = casual, 1 = formal
            "enthusiasm": 0.3,     # 0 = reserved, 1 = excited
            "data_density": 0.9,   # 0 = narrative, 1 = numbers
        },
        "trust_priority": ["on_chain_verification", "block_confirmations", "financial_audit"],
        "copy_rules": {
            "user_term": "network participant",
            "wallet_term": "treasury",
            "action_term": "commit",
        }
    },
    # ... other brands follow same structure
}
```

### What Is Forbidden In A Brand Skin

1. **No new components.** Brand skins do not create components. They restyle existing ones.
2. **No layout changes.** Grid, spacing structure, and page architecture are shared.
3. **No feature toggles.** Features are not brand-specific. Brands can prioritize, not gatekeep.
4. **No custom CSS beyond tokens.** No `.helios-special-card` or `.xxxiii-custom-button`. If a brand needs something new, it goes through the Governor first and becomes a shared component.
5. **No JavaScript overrides.** Behavior is shared. Only data and configuration change.
6. **No conditional brand logic in templates.** Templates render from data. The brand config provides the data. `{% if brand == 'helios' %}` is forbidden — use `{{ module_name }}` instead.

## Brand Skin Validation Checklist

When applying a brand skin, verify:

- [ ] Only the 6 allowed layers changed (palette, emphasis, copy, naming, voice, trust)
- [ ] No new CSS classes or components created
- [ ] No layout or structure modifications
- [ ] No feature gating or brand-specific conditionals
- [ ] Theme file only overrides `--sv-*` custom properties
- [ ] Module names are configured via data, not hardcoded
- [ ] Voice personality is configured, not coded
- [ ] All pages still pass the Builder checklist under the new skin
