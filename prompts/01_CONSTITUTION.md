# MASTER CONSTITUTION

> All systems must obey one constitution, pass one governor, and be built through one shared shell.

This is the highest law. Every AI agent, every developer, every design decision, every product across every brand answers to this document first. If something conflicts with the Constitution, the Constitution wins.

---

## Article I — Identity

UnyKorn is a sovereign technology company.

It builds and operates premium digital systems that move money, create assets, generate proof, and scale trust — across multiple brands, surfaces, and devices.

It is not a design agency. It is not a startup incubator. It is not a content company.

It is an **operating company** that ships revenue-producing systems under a shared technical shell.

---

## Article III — What Every System Must Feel Like

1. **Premium** — Not cheap, not flashy, not busy. Every pixel earns its place.
2. **Minimal** — Less is always the answer until it hurts function.
3. **Alive** — Subtle motion, live data, status indicators. Never frozen.
4. **Intelligent** — AI is behavior, not chrome. The system adapts; it doesn't announce.
5. **Trustworthy** — Proof is visible. Receipts are real. Verification is one click away.
6. **Self-adjusting** — Responsive across devices without separate builds or hacks.
7. **Fast** — Perception of speed matters. Skeleton states. Progressive reveal. No spinners.

---

## Article IV — What Every System Is Allowed To Do

- Move money (accept payments, settle transactions, distribute rewards)
- Create assets (mint tokens, certificates, NFTs, gold-backed instruments)
- Generate proof (on-chain receipts, audit logs, verification surfaces)
- Build trust (KYC flows, identity verification, transparent treasury)
- Scale networks (referral propagation, member nodes, viral loops)
- Serve intelligence (AI advisors, analytics, predictions, automation)
- Host commerce (creator storefronts, marketplace, digital goods)

---

## Article V — What Every System Must Never Become

1. **Confusing** — If a user doesn't understand what to do in 3 seconds, the page has failed.
2. **Cluttered** — If there are more than 3 actions visible at once, the page has failed.
3. **Decorative** — If a visual element doesn't serve function, conversion, or trust, it gets cut.
4. **Fragmented** — If a system feels like a different product from the others, the shell is broken.
5. **Slow** — If a page takes more than 2 seconds to feel ready, the architecture has failed.
6. **Dishonest** — If data is faked, mocked without disclosure, or misleading, the system is corrupt.
7. **Bloated** — If a feature exists that fewer than 10% of users will touch, it should be benched or gated.

---

## Article VI — The Simplicity Rule

Every product, every page, every feature must pass this test:

> **Can a first-time user complete the primary action within 30 seconds of arriving?**

If no: simplify until yes.  
If still no: the feature is too complex to ship. Bench it.

Complexity is earned through progressive reveal, not imposed on arrival.

---

## Article VII — The Money Rule

Every system must answer exactly one of these questions with "yes":

1. **Does it make money?** (Direct revenue: subscriptions, transactions, fees)
2. **Does it save money?** (Automation, efficiency, reduced support load)
3. **Does it build trust?** (Proof, verification, transparency → enables future revenue)
4. **Does it scale the network?** (Viral loops, referrals, propagation → enables future revenue)

If the answer to all four is "no" — **do not build it.**

What never changes: **the shell, the logic, the structure, the motion, the proof language.**

## Article IX — Decision Hierarchy

When any conflict arises — between brands, features, timelines, or opinions — resolve using this order:

1. **Does it violate the Constitution?** → Kill it.
2. **Does it fail the Governor?** → Bench it.
3. **Does it break the shared shell?** → Redesign it.
4. **Does it pass the Money Rule?** → Build it.
5. **Is it top-2 best practice for this problem?** → Ship it.
6. **Everything else** → Backlog.

## Article XI — The Forward Doctrine

**Even when the backend is complex, the product must feel simple, clear, premium, and easy to trust.**

This is the permanent operating rule:

- All new systems follow this doctrine
- All redesigns move toward this doctrine
- All old systems get simplified toward this doctrine over time
- All features must justify themselves against this doctrine

For every system, ask:

1. **What is the simple promise?** — One sentence only.
2. **What is the one main thing the user can do?** — Not five. One.
3. **What is the one reason they trust it?** — Proof, reserve, audit, live state.
4. **What is the one next action?** — Join, activate, verify, buy, share.
5. **What complexity can be hidden until later?** — Everything non-essential.

The two-layer standard:

| Layer | Rule |
|-------|------|
| **Public layer** | Simple, elegant, minimal, premium, obvious |
| **System layer** | Powerful, modular, scalable, compliant, adaptive |

> **We accept backend complexity when necessary, but we never ship front-end confusion.**

> **Everything forward must simplify the experience, sharpen the offer, and strengthen the money path.**
